# Архитектура backend

## 1. Общая идея

Backend проекта строится как модульный монолит.

Это значит:
- приложение остается одним backend-сервисом;
- код делится по бизнес-доменам;
- внутри каждого домена есть своя внутренняя структура;
- инфраструктурные части отделяются от бизнес-логики.

Такой подход выбран для того, чтобы проект было проще развивать, читать и рефакторить по мере роста.

## 2. Базовая структура проекта

```text
src/
  auth/
  users/
  projects/
  statuses/
  sprints/
  issues/
  comments/

  infrastructure/
    prisma/
    config/

  common/
    filters/
    interceptors/
    pipes/
```

### Смысл структуры

- `auth` — аутентификация, access/refresh token, guard, текущий пользователь, сессии.
- `users` — логика пользователей, профиль и связанные пользовательские операции.
- `projects` — проекты и участники проектов.
- `statuses` — колонки проекта и связанные правила board/backlog.
- `sprints` — спринты и их жизненный цикл.
- `issues` — задачи, backlog и движение задач.
- `comments` — комментарии к задачам.
- `infrastructure` — технические зависимости приложения, например Prisma и конфигурация.
- `common` — общие NestJS-части, которые реально используются в нескольких модулях.

## 3. Структура внутри модуля

Каждый модуль строится по слоям внутри своей папки.

Пример:

```text
src/auth/
  controllers/
  services/
  repositories/
  dto/
  responses/
  guards/
  strategies/
  decorators/
  auth.module.ts
```

### Роли папок

- `controllers` — HTTP-вход в модуль.
- `services` — бизнес-логика и use cases.
- `repositories` — доступ к данным, если Prisma-логика становится большой и повторяемой.
- `dto` — входные модели запросов и валидация.
- `responses` — модели ответов API.
- `guards`, `strategies`, `decorators` — security и вспомогательные части, связанные с модулем.

Не обязательно создавать все подпапки сразу. Если модуль пока маленький, структура может быть проще. Новые папки добавляются по мере роста логики.

## 4. Базовые правила архитектуры

1. Проект делится по доменам, а не по глобальным папкам `controllers/services/repositories` на весь backend.
2. Контроллеры должны быть тонкими и не содержать бизнес-логику.
3. Бизнес-логика должна жить в сервисах.
4. Техническая логика базы данных и конфигурации не должна смешиваться с доменной логикой.
5. Общая папка `common` не должна превращаться в свалку случайных хелперов.
6. Если сервис становится слишком большим, его нужно делить на более узкие части.

## 5. Правила зависимостей

- `AppModule` собирает приложение из модулей.
- Доменные модули могут использовать инфраструктурные модули.
- Один модуль не должен ходить во внутренние файлы другого модуля напрямую.
- Если логика должна использоваться снаружи, модуль должен отдавать ее через свой публичный API.
- `PrismaService` относится к инфраструктуре, а не к конкретному бизнес-модулю.

## 6. Ближайшее направление развития

На текущем этапе архитектура уже разложена по модулям, но внутри модулей еще можно усиливать границы.

Ближайшие шаги:
- разгрузить `AuthService`, разделив крупную логику на более мелкие сервисы;
- добавлять `repositories` там, где запросы к Prisma становятся большими или повторяются;
- развивать новые доменные модули в той же структуре;
- не допускать появления “бог-сервисов” и слишком жирных контроллеров.

## 7. Итог

Базовая архитектура backend сейчас такая:
- модульный монолит;
- доменные папки в `src`;
- слои внутри каждого доменного модуля;
- инфраструктура отдельно от бизнес-логики.

Фактически в кодовой базе уже присутствуют модули:
- `auth`
- `users`
- `projects`
- `statuses`
- `sprints`
- `issues`
- `comments`

Это не финальная архитектура “на все времена”, а базовый рабочий каркас, который должен сохранить проект понятным по мере роста.
