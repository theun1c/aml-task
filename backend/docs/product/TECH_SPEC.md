## Техническое задание: Task Manager (гибрид Jira + Trello)

### 1. Цель и назначение продукта

1.1. Назначение
Разработать внутренний инструмент для управления задачами и проектами, сочетающий структурированность Jira (проекты, бэклог, спринты) с гибкостью Trello (Kanban-доска с кастомизируемыми колонками и Drag-n-Drop).

1.2. Целевая аудитория
- Внутренние команды разработки
- Продуктовые команды
- Маркетинговые и операционные подразделения компании

1.3. Ключевые преимущества
- Единое хранилище задач (бэклог) с возможностью планирования по спринтам
- Гибкая доска с настраиваемыми колонками под любой процесс
- Простой и интуитивный интерфейс (минимальный порог входа)

---

### 2. Подход к разработке (MVP + расширение)

Проект реализуется в два этапа:

2.1. MVP (Minimum Viable Product)
Функциональный минимум, позволяющий командам начать работу. Срок реализации — 3-4 недели.

2.2. Пост-MVP (расширение)
Функционал, который проектируется архитектурно, но реализуется после получения обратной связи от пользователей.

---

### 3. Функциональные требования

3.1. MVP

3.1.1. Пользователи и авторизация
- Регистрация по email и паролю
- Вход в систему (email/пароль)
- Просмотр и редактирование профиля (имя, email)

3.1.2. Проекты
- Создание проекта с указанием названия и уникального ключа
- Просмотр списка проектов, в которых пользователь состоит
- При создании проекта автоматически создаются три стандартные колонки: "To Do", "In Progress", "Done"
- Владелец проекта — пользователь, создавший проект

3.1.3. Участники проекта
- Добавление участников в проект по email
- Удаление участников из проекта
- Отображение списка участников
- Две роли: владелец (полный доступ) и участник (ограниченные права)

3.1.4. Бэклог (Backlog)
- Список всех задач проекта, не добавленных ни в один спринт
- Ручная сортировка задач перетаскиванием (Drag-n-Drop)
- Создание новой задачи прямо в бэклоге (название, исполнитель, тип)
- Массовый выбор задач и добавление их в спринт

3.1.5. Спринты (Sprints)
- Создание спринта с указанием названия
- Планирование спринта: интерфейс с двумя колонками (бэклог слева, спринт справа), задачи переносятся перетаскиванием
- Запуск спринта (доступен, если в спринте есть хотя бы одна задача)
- Завершение спринта: все незавершенные задачи автоматически возвращаются в бэклог
- Ограничение: на проект может быть только один активный спринт

3.1.6. Доска (Kanban Board)
- Отображается для активного спринта
- Задачи группируются по колонкам, соответствующим статусам
- Перетаскивание задач между колонками (Drag-n-Drop) с автоматическим обновлением статуса
- Каждая карточка задачи отображает название, тип и исполнителя

3.1.7. Управление колонками (только для владельца проекта)
- Добавление новой колонки с произвольным названием
- Переименование существующей колонки
- Удаление колонки (с подтверждением, задачи переносятся в первую колонку)
- Изменение порядка колонок перетаскиванием

3.1.8. Задачи (Issues)
- Создание задачи (название, описание, тип, исполнитель)
- Редактирование задачи (название, описание, исполнитель, тип)
- Просмотр деталей задачи в модальном окне
- Типы задач: "Задача" и "Ошибка" (визуально различаются иконками)
- Статус задачи автоматически соответствует колонке на доске

3.1.9. Комментарии
- Добавление комментариев к задаче
- Удаление своих комментариев
- Отображение автора, текста и даты комментария

3.2. Расширение (пост-MVP)

| Модуль | Функционал |
|--------|-----------|
| **Задачи** | Story Points, приоритеты, дедлайны, вложения (файлы), Rich Text Editor, история изменений, подзадачи, эпики |
| **Спринты** | Даты начала/окончания, диаграмма сгорания задач (burndown chart), автоматическое завершение по дате |
| **Уведомления** | Email-уведомления, push-уведомления, in-app уведомления, настройка подписки |
| **Поиск и фильтры** | Глобальный поиск по всем задачам, фильтрация по исполнителю, статусу, проекту, типу |
| **Роли и права** | Администратор системы, наблюдатели (только чтение), кастомные роли с гибкими правами |
| **Экспорт** | Экспорт задач в CSV/Excel |
| **Интеграции** | Webhooks, интеграция со Slack и Telegram |
| **Метки/теги** | Создание меток, цветовая маркировка, фильтрация по меткам |
| **API** | Публичное API с документацией OpenAPI для внешних интеграций |

### 4. Нефункциональные требования

4.1. Производительность
- Загрузка страницы проекта — не более 2 секунд
- Drag-n-Drop операции должны выполняться без видимых задержек (отклик < 100 мс)
- Поддержка одновременной работы до 50 пользователей на проекте

4.2. Надежность
- Система должна корректно обрабатывать ошибки сети с откатом изменений
- Данные не должны теряться при разрыве соединения во время операций

4.3. Безопасность
- Пароли хранятся в зашифрованном виде (bcrypt)
- JWT-токены с ограниченным сроком действия
- Доступ к задачам только участников соответствующего проекта

4.4. Масштабируемость
- Архитектура БД спроектирована с учетом возможности горизонтального масштабирования
- API спроектировано как stateless, что позволяет запускать несколько экземпляров приложения

4.5. Удобство использования (UX)
- Минимальное количество кликов для основных операций (создание задачи — 2 клика)
- Визуальная обратная связь на все действия (загрузка, успех, ошибка)
- Подтверждение опасных действий (удаление колонки, завершение спринта)

### 5. Технологический стек

5.1. Frontend
| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| Фреймворк | React 18+ | Популярность, экосистема, компонентный подход |
| Управление состоянием | React Query + Zustand | React Query для серверных данных, Zustand для UI-состояния |
| Drag-n-Drop | @dnd-kit | Производительный, гибкий, современный |
| Стилизация | Tailwind CSS | Быстрая разработка, удобство поддержки |
| UI-компоненты | shadcn/ui или Radix UI | Доступность, кастомизация, качественная база |
| Роутинг | React Router v6 | Стандарт де-факто для React |

5.2. Backend
| Компонент | Технология | Обоснование |
|-----------|------------|-------------|
| Runtime | Node.js 20+ | Единый язык с фронтендом, большая экосистема |
| Фреймворк | NestJS | Масштабируемость, модульная архитектура, TypeScript-нативный |
| ORM | Prisma | TypeScript-first, автодополнение, удобные миграции |
| База данных | PostgreSQL | Надежность, поддержка JSONB для гибких настроек |
| Авторизация | JWT | Stateless, простота реализации |
| Документация API | Swagger/OpenAPI | Автоматическая генерация, удобство тестирования |

5.3. Инфраструктура

| Компонент | Рекомендация | Альтернатива |
|-----------|--------------|--------------|
| Хостинг фронтенда | Vercel | Netlify, AWS S3 + CloudFront |
| Хостинг бэкенда | Railway | Heroku, AWS ECS |
| База данных | Supabase (PostgreSQL) | AWS RDS, DigitalOcean Managed DB |
| Файловое хранилище (post-MVP) | AWS S3 | Cloudflare R2, MinIO |

### 6. Архитектура базы данных (ключевые сущности)

Ниже приведено описание основных таблиц и связей. Полная схема с полями фиксируется в документации для разработчиков.

6.1. Пользователи (users)
- Хранит учетные записи: email, пароль, имя, аватар
- Расширяемые настройки в формате JSON

6.2. Проекты (projects)
- Основная единица группировки задач
- Содержит название, уникальный ключ, описание, владельца
- Поддерживает архивацию

6.3. Участники проектов (project_members)
- Связующая таблица между пользователями и проектами
- Фиксирует роль участника (владелец, участник)

6.4. Статусы (statuses)
- Колонки на Kanban-доске
- Привязаны к конкретному проекту
- Имеют порядок отображения, категорию (todo/active/done), цвет

6.5. Спринты (sprints)
- Итерации планирования
- Привязаны к проекту, имеют флаг активности
- В пост-MVP добавляются даты и цель спринта

6.6. Задачи (issues)
- Центральная сущность системы
- Связи: проект, спринт, статус, исполнитель, создатель
- Содержит название, описание, тип, позицию для сортировки
- Поля для пост-MVP (story points, приоритет, дедлайн) заложены и могут быть nullable

6.7. Комментарии (comments)
- Обсуждения внутри задачи
- Привязка к задаче и автору

6.8. Вспомогательные таблицы (пост-MVP)
- **История изменений (issue_history)** — аудит всех изменений задачи
- **Вложения (attachments)** — файлы, прикрепленные к задаче
- **Метки (labels, issue_labels)** — теги для классификации задач

### 7. API (основные эндпоинты)

API проектируется как RESTful. Ниже приведен список основных групп эндпоинтов. Детальная спецификация с методами и форматами данных будет подготовлена на этапе проектирования.

| Группа | Описание | Ключевые операции |
|--------|----------|-------------------|
| **Auth** | Авторизация и регистрация | Регистрация, вход, получение текущего пользователя |
| **Users** | Управление пользователями | Поиск по email, получение профиля |
| **Projects** | Управление проектами | CRUD проектов, получение списка, добавление/удаление участников |
| **Statuses** | Управление колонками | CRUD статусов, изменение порядка |
| **Sprints** | Управление спринтами | CRUD спринтов, запуск, завершение, управление составом |
| **Issues** | Управление задачами | CRUD задач, изменение статуса, обновление позиции |
| **Comments** | Управление комментариями | Добавление, редактирование, удаление |

### 8. Критерии приемки MVP

8.1. Основные сценарии

| № | Сценарий | Ожидаемый результат |
|---|----------|---------------------|
| 1 | Пользователь регистрируется и входит в систему | Успешный вход, отображение дашборда проектов |
| 2 | Создание проекта с названием и ключом | Проект создан, создатель назначен владельцем, созданы три колонки по умолчанию |
| 3 | Добавление участника по email | Участник появляется в списке проекта |
| 4 | Создание задачи в бэклоге | Задача отображается в списке бэклога |
| 5 | Создание спринта и добавление в него задачи | Задача перенесена из бэклога в спринт |
| 6 | Запуск спринта | Появляется доска, задачи отображаются в колонках |
| 7 | Перетаскивание задачи между колонками на доске | Статус задачи обновлен, после обновления страницы задача остается в новой колонке |
| 8 | Добавление новой колонки владельцем проекта | Колонка появляется на доске |
| 9 | Переименование колонки | Название колонки обновлено на доске |
| 10 | Изменение порядка колонок | Колонки отображаются в новом порядке |
| 11 | Завершение спринта | Задачи в незавершенных статусах возвращены в бэклог, доска скрыта |
| 12 | Открытие задачи, добавление комментария | Комментарий сохраняется и отображается |
| 13 | Редактирование задачи (смена исполнителя, типа) | Изменения сохранены и отображаются |

8.2. Технические критерии
- Время отклика API на основные операции не превышает 500 мс (p95)
- Приложение корректно работает в актуальных версиях Chrome, Firefox, Safari
- Отсутствие критических багов, влияющих на сохранность данных
- Код покрыт базовыми автотестами (не менее 60% для бизнес-логики)

### 9. Ограничения MVP

Следующий функционал **не входит** в объем MVP, но закладывается архитектурно для будущего расширения:

| Область | Исключенный функционал |
|---------|------------------------|
| Задачи | Story Points, приоритеты, дедлайны, вложения файлов, Rich Text Editor, история изменений, подзадачи, эпики |
| Спринты | Даты начала/окончания, диаграмма сгорания задач |
| Уведомления | Email, push, любые уведомления кроме визуальных toasts |
| Поиск | Глобальный поиск, сложные фильтры |
| Роли | Администратор системы, наблюдатели, гибкие права |
| Интеграции | Webhooks, внешние сервисы |
| Метки | Любая функциональность тегов |
| Экспорт | Выгрузка данных |
| API | Публичная документация, доступ для внешних систем |

### 10. Риски и пути их снижения

| Риск | Вероятность | Влияние | Способ снижения |
|------|-------------|--------|-----------------|
| Сложность Drag-n-Drop на доске с большим количеством задач | Средняя | Высокое | Использовать проверенную библиотеку @dnd-kit, внедрить виртуализацию списков |
| Неоднозначность требований к статусам и колонкам | Низкая | Среднее | Жестко зафиксировать MVP-набор, гибкость реализовать в пост-MVP |
| Задержки при интеграции авторизации | Низкая | Среднее | Использовать стандартные JWT-механизмы, не реализовывать сложные схемы восстановления пароля в MVP |
| Усложнение архитектуры из-за закладок под пост-MVP | Средняя | Среднее | Четко разделить: MVP-поля обязательны, пост-MVP-поля nullable без бизнес-логики |
| Недостаточное тестирование Drag-n-Drop | Средняя | Высокое | Выделить отдельный этап на тестирование всех сценариев перетаскивания на разных устройствах |

---

### 11. Критерии успеха проекта

MVP считается успешно реализованным, если:

1. Все сценарии из раздела 8.1 успешно протестированы и работают
2. Команда из 5-10 человек может вести свои проекты в системе без сбоев в течение недели пилотной эксплуатации
3. Время создания задачи от открытия формы до сохранения не превышает 30 секунд
4. Отсутствуют критические баги, приводящие к потере данных
5. Обратная связь от пилотных пользователей собрана и задокументирована для планирования пост-MVP
