# CONTEXT.md

## О проекте
Проект: AML Task Manager (гибрид Jira + Trello)

Цель MVP:
Сделать внутренний инструмент для управления проектами и задачами, который сочетает:
- backlog и sprint planning как в Jira;
- kanban-доску и drag-n-drop как в Trello.

Основной акцент MVP:
- быстрое создание и движение задач;
- проектная работа в небольших командах;
- простая и понятная модель прав;
- минимально достаточная архитектура без перегруза.

## Подход к разработке
Проект разрабатывается поэтапно:
1. MVP
2. post-MVP расширение

Важно:
- не реализовывать post-MVP функциональность заранее;
- но учитывать расширяемость схемы БД и API.

## Бизнес-правила MVP

### Пользователи
- регистрация по email + password
- логин по email + password
- активные refresh-сессии хранятся в `user_sessions`

### Проекты
- проект имеет owner
- при создании проекта создаются статусы:
  - To Do
  - In Progress
  - Done
- архивированный проект остается читаемым для участников, но все project-scoped mutation endpoint'ы работают в режиме read-only и возвращают `409 Conflict`

### Участники
- роли:
  - owner
  - member
- только участники проекта могут видеть его данные

### Спринты
- у проекта может быть только один активный спринт
- planned sprint можно читать, обновлять и удалять
- удаление planned sprint возвращает все его задачи в backlog
- завершение спринта возвращает незавершенные задачи в backlog

### Задачи
- типы задач:
  - task
  - bug
- в БД тип хранится через `issue_types`, а порядок через `rank_position`
- статус задачи определяется колонкой
- задача может быть без спринта (backlog)
- soft-delete issues сохраняет `issue_number` и не ломает историю нумерации внутри проекта
- смена статуса и reorder внутри sprint разрешены только для active sprint board, но no-op сценарии остаются допустимыми

### Профиль пользователя
- доступны `GET /users/me`, `PATCH /users/me`
- доступен `GET /users/search?email=...` для поиска активных пользователей по email-фрагменту

### Комментарии
- пользователь может удалять и редактировать только свои комментарии
- comment content trim-ится и whitespace-only значения считаются невалидными

## Архитектурные ориентиры backend

### Общие принципы
- NestJS modules по доменам
- controllers тонкие
- services содержат бизнес-логику
- доступ к БД через Prisma
- DTO валидируются
- права доступа проверяются явно
- публичный REST API использует `snake_case`, согласованный с БД и Prisma-схемой

### Предпочтительная структура модулей
- auth
- users
- projects
- project-members
- statuses
- sprints
- issues
- comments

### Текущее состояние документации
- детальные feature docs сейчас есть для `auth`, `issues`, `users`, `projects`, `statuses`, `sprints`, `comments`;
- документация по этим модулям должна синхронизироваться вместе с кодом, особенно при изменении публичных endpoint'ов и бизнес-правил архивации;
- перед доработкой модулей нужно сверять фактическое поведение кода с `TECH_SPEC.md` и фиксировать расхождения явно.

## Ограничения MVP
Не реализуем в MVP:
- story points
- priorities
- deadlines
- attachments
- labels
- sub-tasks
- epics
- notifications
- global search
- custom roles
- integrations
- export

---

## Что важно при проектировании
1. Не усложнять.
2. Не писать лишнюю абстракцию “на будущее”.
3. Но:
   - учитывать nullable-поля под post-MVP;
   - не ломать API будущим расширениям;
   - не завязывать бизнес-логику на UI.

---

## Что особенно важно для агента
Разработчик изучает NestJS и TypeScript.
Нужен формат помощи через:
- объяснение,
- декомпозицию,
- точечные примеры,
- ревью,
- помощь в проектировании.

Нежелательно:
- генерировать большие куски готового кода без прямого запроса;
- скрывать сложность за магией фреймворка без объяснения.
